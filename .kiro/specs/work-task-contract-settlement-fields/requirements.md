# 業務リスト「契約決済」タブのフィールド追加・変更

## 1. 概要

業務リスト詳細モーダルの「契約決済」タブにおいて、以下の変更を実施する：

1. 「契約形態」フィールドの上に「売買資料ドライブ」フィールドを追加
2. 「広瀬さんへ依頼（売買契約関連）」フィールドの同期問題を調査・修正（AA2022で同期されていない）
3. 「社員が契約書作成」フィールドをスタッフシートの「通常=true」のイニシャルを表示するように変更
4. 「作業内容」フィールドの下に「作業完了コメント」フィールドを追加
5. 「作業完了コメント」の下に条件付きフィールドを追加：
   - 「広瀬さんへ依頼（売買契約関連）」が埋まっている場合：「作業完了チャット（廣瀬）」を表示
   - 「CWへ依頼（売買契約関連）」が埋まっている場合：「CWへ完了チャット」を表示
6. 「作業完了チャット」の下に「完了コメント（売買関連）」フィールドを追加
7. 「添付資料印刷」フィールドを削除
8. 「添付資料準備納期」フィールドを削除
9. 「添付資料完了」フィールドを削除
10. **チャット送信フィールドを物件担当者のみに変更**：
    - 「国広とチャット」「山本へチャット送信」「裏へチャット送信」「角井へチャット送信」を削除
    - 「{担当者イニシャル}へチャット送信」フィールドを追加（Y/Nボタン）
    - チャットアドレスはスタッフシートから取得
    - 例：AA2022の担当が「U」の場合、「Uへチャット送信」のみ表示
11. **売買契約確認フィールドの位置を変更**：
    - 「売買契約確認」を「売買契約締め日」の直後に移動

## 2. 背景

### 2.1 現在の問題
- 「広瀬さんへ依頼（売買契約関連）」が物件AA2022で同期されていない
- 「社員が契約書作成」が固定のオプション（ASSIGNEE_OPTIONS）を使用している
- 「添付資料印刷」「添付資料準備納期」「添付資料完了」フィールドが不要になった
- 作業完了の管理フィールドが不足している
- **チャット送信フィールドが全スタッフ分表示されている**（国広、山本、裏、角井）
- **物件担当者のみのチャット送信フィールドが必要**

### 2.2 改善の目的
- 売買資料ドライブのURLを管理できるようにする
- スタッフシートの「通常=true」のスタッフのみを表示して、選択肢を動的にする
- 作業完了コメントと完了チャットを管理できるようにする
- 依頼先に応じて適切な完了チャットフィールドを表示する
- **物件担当者のみのチャット送信フィールドを表示する**
- **チャットアドレスをスタッフシートから動的に取得する**

## 3. ユーザーストーリー

### 3.1 売買資料ドライブの追加
**As a** 業務担当者  
**I want to** 売買資料ドライブのURLを業務リストに登録できる  
**So that** 売買契約関連の資料を一元管理できる

**受け入れ基準**:
- 「契約決済」タブの「契約形態」フィールドの上に「売買資料ドライブ」フィールドが表示される
- URLフィールドとして実装され、リンクをクリックすると新しいタブで開く
- スプレッドシートの対応するカラムと同期される

### 3.2 広瀬さんへ依頼の同期修正
**As a** 業務担当者  
**I want to** 「広瀬さんへ依頼（売買契約関連）」が正しく同期される  
**So that** 物件AA2022でも正しく表示・編集できる

**受け入れ基準**:
- 物件AA2022で「広瀬さんへ依頼（売買契約関連）」が正しく表示される
- スプレッドシートとの同期が正常に動作する
- 同期されない原因を特定して修正する

**調査結果**:
- スプレッドシートには「N」が入っている
- データベースには「FALSE」（文字列）として保存されている
- `EditableYesNo`コンポーネントが「FALSE」文字列を正しく認識していなかった
- 修正：`EditableYesNo`コンポーネントを拡張して、以下の全ての形式を認識するようにした：
  - Y/N（文字列）
  - TRUE/FALSE（文字列、大文字）
  - true/false（文字列、小文字）
  - true/false（ブール値）
  - 大文字小文字を区別しない文字列比較

### 3.3 社員が契約書作成の動的選択肢
**As a** 業務担当者  
**I want to** 「社員が契約書作成」でスタッフシートの「通常=true」のイニシャルのみを選択できる  
**So that** 現在アクティブなスタッフのみが表示される

**受け入れ基準**:
- 「社員が契約書作成」フィールドの選択肢がスタッフシートの「通常=true」のイニシャルになる
- 既存の`activeStaffInitials`を使用する（サイト登録タブと同じロジック）
- 固定のASSIGNEE_OPTIONSは使用しない

### 3.4 作業完了コメントの追加
**As a** 業務担当者  
**I want to** 作業完了のコメントを記録できる  
**So that** 作業完了の詳細を管理できる

**受け入れ基準**:
- 「契約決済」タブの「作業内容」フィールドの下に「作業完了コメント」フィールドが表示される
- 複数行テキストフィールドとして実装される
- スプレッドシートの対応するカラムと同期される

### 3.5 条件付き完了チャットフィールドの追加
**As a** 業務担当者  
**I want to** 依頼先に応じた完了チャットフィールドが表示される  
**So that** 適切な完了報告ができる

**受け入れ基準**:
- 「広瀬さんへ依頼（売買契約関連）」が「Y」の場合、「作業完了チャット（廣瀬）」フィールドが表示される
- 「CWへ依頼（売買契約関連）」が「Y」の場合、「CWへ完了チャット」フィールドが表示される
- 両方が「Y」の場合、両方のフィールドが表示される
- どちらも「Y」でない場合、どちらのフィールドも表示されない
- スプレッドシートの対応するカラムと同期される

### 3.6 完了コメント（売買関連）の追加
**As a** 業務担当者  
**I want to** 売買関連の完了コメントを記録できる  
**So that** 売買契約の完了詳細を管理できる

**受け入れ基準**:
- 「契約決済」タブの「作業完了チャット」フィールドの下に「完了コメント（売買関連）」フィールドが表示される
- 複数行テキストフィールドとして実装される
- スプレッドシートの対応するカラムと同期される

### 3.7 不要フィールドの削除
**As a** 業務担当者  
**I want to** 「添付資料印刷」「添付資料準備納期」「添付資料完了」フィールドが表示されない  
**So that** 不要なフィールドが画面に表示されない

**受け入れ基準**:
- 「契約決済」タブから「添付資料印刷」「添付資料準備納期」「添付資料完了」フィールドが削除される
- データベースのカラムは残す（既存データを保持）
- スプレッドシートとの同期は維持する（削除しない）

### 3.8 物件担当者のチャット送信フィールド
**As a** 業務担当者  
**I want to** 物件担当者のみのチャット送信フィールドが表示される  
**So that** 担当者に直接チャットを送信できる

**受け入れ基準**:
- 「国広とチャット」「山本へチャット送信」「裏へチャット送信」「角井へチャット送信」フィールドが削除される
- 「{担当者イニシャル}へチャット送信」フィールドが表示される（Y/Nボタン）
- 担当者イニシャルは`sales_assignee`フィールドから取得
- チャットアドレスはスタッフシートから取得
- 例：AA2022の担当が「U」の場合、「Uへチャット送信」が表示される
- スプレッドシートの対応するカラムと同期される

### 5.1 システム隔離ルール
- **公開物件サイトには絶対に影響を与えない**
- 業務リスト専用のファイルのみを変更する
- 以下のファイルは変更しない：
  - `backend/api/index.ts`
  - `backend/api/src/services/PropertyListingService.ts`
  - `frontend/src/pages/PublicProperty*.tsx`

### 5.2 スプレッドシートヘッダー保護ルール

## 9. リスク

### 9.1 スプレッドシートのカラム名不一致
- **リスク**: 「売買資料ドライブ」カラムがスプレッドシートに存在しない
- **対策**: スプレッドシートを確認してから実装する

### 9.2 広瀬さんへ依頼の同期問題
- **リスク**: 原因が特定できない可能性がある
- **対策**: スプレッドシートのデータ、カラムマッピング、同期ロジックを順番に確認する

### 9.3 既存データへの影響
- **リスク**: フィールドの削除により既存データが失われる
- **対策**: 「添付資料印刷」「添付資料準備納期」「添付資料完了」はフロントエンドから削除するが、データベースとスプレッドシートの同期は維持する

### 9.4 条件付き表示ロジックの複雑化
- **リスク**: 条件付きフィールドの表示ロジックが複雑になり、バグが発生する可能性がある
- **対策**: 既存の条件付き表示ロジック（サイト登録タブの「サイト登録確認者」）を参考にする
